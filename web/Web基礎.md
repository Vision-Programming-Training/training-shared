# Web アプリケーション基礎レクチャー

この資料は、研修で触るミニEC（注文API）が **「そもそもどういう仕組みで動いているのか」** を、ざっくりつかむための入門です。<br>
コードを直す前に、「ブラウザ」「サーバー」「API」といった言葉が指すものと、それらが**どう繋がって1つのアプリになっているか**の地図を持っておくのが狙いです。

深い理論には踏み込みません。**登場人物と、データが流れる向き**が分かれば十分です。

---

## 1. Web アプリケーションの全体像

普段みなさんが使っているネットショップやSNSは、**1台のパソコンの中だけ**で完結していません。<br>
大きく **2人の登場人物** が、ネットワーク越しに会話することで動いています。

| 登場人物 | ざっくり言うと | この研修での実体 |
|---|---|---|
| **クライアント** | 「お願いする側」。ユーザーが直接触る画面 | ブラウザで動く [training-frontend](../guides/受講者ガイド.md)（HTML + JavaScript） |
| **サーバー** | 「お願いされて仕事をする側」。裏方 | [training-backend](../guides/受講者ガイド.md)（ASP.NET / C#） |

```
[ユーザー]
   │ 画面を操作（クリック・入力）
   ▼
┌──────────────┐        お願い（リクエスト）      ┌──────────────┐
│  クライアント   │ ──────────────────────────▶ │   サーバー     │
│（ブラウザ）     │                              │（バックエンド） │
│  HTML/JS       │ ◀────────────────────────── │  C#            │
└──────────────┘        答え（レスポンス）        └──────┬───────┘
                                                        │ 読み書き
                                                        ▼
                                                  ┌──────────┐
                                                  │   DB      │
                                                  │（SQLite） │
                                                  └──────────┘
```

> ポイントは **「クライアントとサーバーは別物」** ということ。<br>
> 画面（フロント）でボタンを押しても、実際の計算やデータ保存は**別の場所にいるサーバー**がやっています。だからバグも「画面側の問題か / サーバー側の問題か」で切り分けが必要になります。

---

## 2. ブラウザの仕組み

**ブラウザ**（Chrome / Edge / Safari など）は、クライアントの正体です。ざっくり次の3つの仕事をしています。

1. **URL のページを取りに行く**（サーバーにお願いする）
2. **返ってきた HTML / CSS / JavaScript を解釈して、画面を組み立てる**（レンダリング）
3. **ユーザーの操作を受け取り、必要ならまたサーバーにお願いする**

### URL の中身

アドレスバーに入っている文字列（URL）は、「どのサーバーの、どの場所を」を表す住所です。

```
https://example.com:5000/api/orders?status=Paid
└─┬─┘  └────┬────┘└─┬┘└───┬────┘└────┬─────┘
 方式      ホスト     ポート   パス      クエリ（条件）
```

| 部品 | 意味 |
|---|---|
| 方式（スキーム） | `http` / `https`。`https` は通信が暗号化されている |
| ホスト | どのサーバーか（`example.com`、開発中は `localhost`＝自分のPC） |
| ポート | サーバーの「入口の番号」（例: `5000`）。1台で複数のアプリを動かすため |
| パス | サーバーの中のどの機能か（`/api/orders`） |
| クエリ | 追加の条件（`?status=Paid` = 支払い済みだけ） |

> 研修中、バックエンドを起動すると `http://localhost:5000` のような URL が出ます。これは **「自分のPCの中でサーバーが動いていて、5000番の入口で待っている」** という意味です。

### HTML / CSS / JavaScript の役割分担

ブラウザが受け取って画面を作る材料は、役割が分かれています。

| 材料 | 役割 | たとえ |
|---|---|---|
| **HTML** | 構造・中身（見出し・表・ボタン） | 家の骨組み |
| **CSS** | 見た目（色・大きさ・配置） | 内装・ペンキ |
| **JavaScript** | 動き（ボタンを押したら…、サーバーと通信…） | 電気・スイッチ |

このミニECのフロントは HTML + JavaScript でできていて、**「注文ボタンを押したら JavaScript がサーバーに注文をお願いする」** という流れで動いています。

---

## 3. サーバーって何ですか

**サーバー**は、クライアントからの「お願い（リクエスト）」を受け取り、仕事をして「答え（レスポンス）」を返し続けているプログラムです。<br>
「サーバー」という特別なマシンがあるわけではなく、**ずっと起動しっぱなしで、リクエストを待ち受けているプログラム**のことを指します（研修では自分のPCの中で動かします）。

サーバーがやっていること（このミニECの例）:

- 「注文一覧をください」→ **DB から注文を読んで**返す
- 「この内容で注文します」→ **金額を計算し、在庫を減らし、DB に注文を保存**して結果を返す
- 「変なクーポンコードだ」→ **エラーを返す**

> **ロジック（計算・判断）とデータ（DB への保存）は、基本サーバー側にある**、と覚えてください。<br>
> 金額計算のバグ（A1・A7）や在庫のバグ（A2・A4）は、画面ではなく**サーバーのコード**を読むことになります。DB の中身を直接確認する方法は [`../db/SQLite基礎.md`](../db/SQLite基礎.md) を参照。

## 4. HTTP — クライアントとサーバーの共通言語

クライアントとサーバーは **HTTP** というルール（プロトコル）で会話します。<br>
会話は必ず **「リクエスト（お願い）1つ ＋ レスポンス（答え）1つ」** のセットです。クライアントから話しかけ、サーバーが答える。この一往復のくり返しでアプリは動きます。

### HTTP メソッド — 「何をしたいか」

お願いには「何をしたいか」の種類（メソッド）があります。よく使う4つ:

| メソッド | やりたいこと | ミニECの例 |
|---|---|---|
| `GET` | 取得する（読む） | 注文一覧・商品一覧を見る |
| `POST` | 新しく作る | 注文を1件作る |
| `PUT` / `PATCH` | 書き換える | 注文をキャンセル状態にする |
| `DELETE` | 消す | （このアプリではあまり使わない） |

> DB の CRUD（[`../db/SQLite基礎.md`](../db/SQLite基礎.md) §2）と似ています。`GET`＝SELECT、`POST`＝INSERT… とおおまかに対応づくと理解が早いです。

### ステータスコード — 「結果どうだった？」

レスポンスには、**成功したか失敗したかを表す3桁の番号**が必ず付きます。エラーの切り分けで毎回見ることになります。

| 番号帯 | 意味 | 代表例 |
|---|---|---|
| **2xx** | 成功 | `200 OK`（取得成功）/ `201 Created`（作成成功） |
| **4xx** | **お願いした側（クライアント）が悪い** | `400 Bad Request`（入力が変）/ `404 Not Found`（そんなデータない） |
| **5xx** | **サーバー側が壊れた** | `500 Internal Server Error`（サーバー内部でエラー） |

> **4xx と 5xx の違いは超重要**。<br>
> 「存在しない注文IDを開いたら `500`（サーバーエラー）になる」課題（A6）は、本当は **`404`（そんな注文はない）を返すべき**なのに、サーバーが想定外に落ちて `500` を返してしまっている、という話です。番号を見れば「これはサーバーが想定外に壊れているな」と当たりがつきます。

## 5. API と JSON — サーバーとの"データのやり取り"

### API とは

画面（HTML）ではなく、**プログラム同士がデータをやり取りするための"窓口"**を **API** と呼びます。<br>
このミニECのフロントは、サーバーが用意した **注文API**（`/api/orders` など）を叩いて、注文データをもらったり送ったりしています。

「注文一覧API」「注文作成API」のように、**機能ごとに窓口（URL）が分かれている**イメージです。どんな窓口があるかはバックエンドの仕様（[`../plans/`](../README.md) の契約）で決まっています。

### JSON — やり取りに使うデータの形

API でやり取りするデータは、多くの場合 **JSON** という形式です。人にもプログラムにも読める、シンプルなテキストです。

```json
{
  "id": 1,
  "status": "Paid",
  "totalAmount": 1980,
  "items": [
    { "productName": "ノート", "quantity": 2, "unitPrice": 300 },
    { "productName": "マグカップ", "quantity": 1, "unitPrice": 1200 }
  ]
}
```

- `{ }` … ひとまとまりのデータ（オブジェクト）。`"名前": 値` の集まり
- `[ ]` … 同じ種類のものが複数並ぶ（配列）。ここでは注文明細が2件
- 値は 文字列（`"Paid"`）・数値（`1980`）・入れ子など

> この JSON の中身は、[`../db/データモデル.md`](../db/データモデル.md) の `Order` / `OrderItem` とほぼ対応します。<br>
> つまり **DB に入っているデータ → サーバーが JSON に組み立てて返す → ブラウザが画面に表示**、という流れ。金額がずれるバグ（A1/A7）は「DB の値」「サーバーが返す JSON の値」「画面の表示」のどこでずれているかを追うことになります。

## 6. 通しで見る — 「注文ボタンを押してから画面が変わるまで」

ここまでの登場人物を、1つの操作でつなげてみます。

```
① ユーザーが「注文する」ボタンをクリック
        │
② ブラウザの JavaScript が、注文内容を JSON にして
   POST /api/orders でサーバーにリクエスト
        │
③ サーバー（C#）が受け取り…
     ・入力チェック（クーポンは正しい？在庫は足りる？）
     ・金額を計算（小計 → クーポン → 税込・四捨五入）
     ・DB（SQLite）に注文を保存 / 在庫を減らす
        │
④ サーバーが結果を JSON ＋ ステータスコード（201 など）で返す
        │
⑤ ブラウザが受け取った JSON を読んで、画面を「注文完了」に更新
```

研修の課題は、この ①〜⑤ の**どこか**が期待とズレている、という形で現れます。<br>
だから最初にやるのは「**症状はどの登場人物のところで起きている？**」の切り分けです。

| 症状 | だいたいの疑い場所 |
|---|---|
| 表示が崩れる・ボタンが効かない | ② フロント（HTML/JS） |
| 金額・在庫の計算がおかしい | ③ サーバーのロジック |
| 保存したはずのデータが変・戻らない | ③〜 サーバー ↔ DB |
| `500` エラーで落ちる | ③ サーバーが想定外で壊れている |

## 7. 用語まとめ

| 用語 | ひとことで |
|---|---|
| クライアント | お願いする側。ユーザーが触る画面（＝ブラウザ） |
| サーバー | お願いされて仕事をする裏方。ずっと待ち受けているプログラム |
| ブラウザ | URL のページを取りに行き、HTML/CSS/JS を画面に組み立てるアプリ |
| フロントエンド | クライアント側で動く部分（見た目・操作）。この研修は HTML + JS |
| バックエンド | サーバー側で動く部分（計算・保存）。この研修は C# |
| HTTP | クライアントとサーバーの共通言語。リクエストとレスポンスの一往復 |
| メソッド | お願いの種類（GET / POST / PUT / DELETE） |
| ステータスコード | 結果を表す3桁の番号（2xx 成功 / 4xx 依頼側ミス / 5xx サーバー故障） |
| API | プログラム同士がデータをやり取りする窓口 |
| JSON | やり取りに使う、人にも読めるデータの形式 |
| DB | データを消えないようにしまう入れ物（[`../db/SQLite基礎.md`](../db/SQLite基礎.md)） |

---

## 8. まとめ

- Web アプリは **クライアント（ブラウザ）** と **サーバー（バックエンド）** の2人の会話でできている。
- 会話は **HTTP**。「リクエスト → レスポンス」の一往復のくり返し。
- **見た目・操作はフロント、計算・保存はサーバー**。DB はさらにその裏。
- レスポンスの **ステータスコード** を見れば、成功か・誰のせいの失敗かが分かる。
- API では **JSON** でデータをやり取りし、その中身は DB のデータモデルとほぼ対応する。

> この資料は「全体像の地図を持つ」までが目標です。実際にどの URL でどんな JSON をやり取りしているかは、ブラウザの開発者ツール（F12）の「Network」タブで**本物の通信**を覗いてみると一気に腑に落ちます。
