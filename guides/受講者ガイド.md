# ミニEC 既存改修トレーニング — 受講者ガイド

ようこそ。この研修は **「ゼロからアプリを作る」のではなく、すでに動いている既存アプリに手を入れて直す」** 練習です。配属先では、他人が書いた・もう動いているコードを読んで直す場面がほとんど。その地力を、小さなミニEC（ネットショップの注文API）で鍛えます。

特定の言語を覚えるのが目的ではありません。**どの現場でも効く次の力**を身につけます。

- 他人が書いた、もう動いているコードを読んで全体像をつかむ
- 「直したい挙動はどのファイル・どの層にある？」を探し当てる
- 既存の作法に合わせて、**最小限だけ**直す（自己流で書き換えない）
- テストと git で「壊していないこと」を自分で担保する
- Issue / PR / レビューのチーム開発フローに乗る

---

## 1. 最初にやること（環境構築）

リポジトリは2つあります。両方を clone してください。

```bash
# サーバー（研修の主戦場）
git clone <training-backend-csharp の URL>
cd training-backend-csharp
dotnet run --project src        # ビルド → 起動 → DBに初期データ投入まで自動

# フロント（別ターミナル）
git clone <training-frontend の URL>
cd training-frontend
# index.html を簡易サーバーで開く（VS Code の Live Server 拡張など）
```

- サーバーが起動したら、ブラウザで **`http://localhost:<ポート>/swagger`** を開く。フロント無しでも API を叩いて挙動を確認できます。
- フロントの `js/config.js` に、サーバーの URL（例 `http://localhost:5000`）が書いてあります。
- フロントとサーバーは別物（別オリジン）なので、繋ぐのに **CORS** という設定が要ります。これは**設定済み**で渡しているので、今は気にしなくてOK（「クラサバは別物だから繋ぐのに一手間いる」という事実だけ頭の隅に）。

> うまく起動しないときは講師に相談。ここで時間を使うのが目的ではありません。

## 2. アプリの地図（どこに何があるか）

サーバーは **3つの層**に分かれています。この役割分担を最初に頭に入れておくと、「直す場所」が早く見つかります。

```
src/
├── Controllers/   ← APIの入口。リクエストを受けるだけ（薄い）。ロジックは書かない
│   ├── ProductsController.cs
│   └── OrdersController.cs
├── Services/      ← 業務ロジックの中心。★直す対象の多くはここ★
│   ├── OrderService.cs      （在庫・クーポン・注文作成/キャンセル）
│   └── PricingService.cs    （金額計算：小計→割引→税込）
├── Repositories/  ← DB(データベース)とのやりとり
├── Entities/      ← データの形（商品・注文・明細・クーポン）
├── Dtos/          ← APIが返す/受け取るデータの形
├── Data/          ← DBの初期データ（SeedData.cs）
└── Middleware/    ← 例外を HTTP のエラー応答に変換する場所
```

**覚えておく原則**: Controller は薄く、**業務の判断は Service にある**。「金額がおかしい」「在庫がおかしい」なら、まず `Services/` を見る。

### 金額の計算ルール（重要）

```
1. 税抜小計  = Σ(単価 × 数量)
2. クーポン  = 定額(◯◯円引き) または 定率(◯%引き)。引きすぎて0円未満なら0円
3. 税込合計  = (割引後) × 1.10 を、1円未満は四捨五入（.5は切り上げ）
```

検証用の値（覚えておくと原因特定が速い）:
- ノート2 + マグ1（割引なし）= **1980円**
- Tシャツ1 + ペン3 − `WELCOME500`(500円引き) = **2695円**
- トートバッグ1 − `SALE10`(10%引き) = **1782円**

## 3. テストと git — 「直った」「壊してない」の目印

```bash
dotnet test     # サーバーのテストを全部走らせる
```

- テストが **緑（成功）= 仕様どおり動いている**目印。**赤（失敗）= どこか壊れている**目印。
- 課題に取りかかったら、まず `dotnet test` を実行。**どのテストが赤いか**が、症状の正体を教えてくれることが多いです。
- 直すときは新しいブランチを切る（`CONTRIBUTING.md` の命名規則に従う）。直し終えたら全テストが緑であることを確認 → PR。

> **テストが赤いまま放置しない。** 自分の変更で別のテストを赤くしていないか、必ず最後に全テストを見る。

## 4. 課題の進め方（毎回この順番で）

1. **課題ブランチをチェックアウトする** 課題番号に応じた `challenge/...`をチェックアウトする。
2. **（バグの場合）バグを再現させる。** 原因解明のためにはまず再現させることが大事。
3. **`dotnet test` を走らせる。** 赤いテストがあれば、その期待値（正しい値）と実際の値を比べる。
4. **原因を解析する。** いきなり書き換えない。まず「なぜこの行でそうなるか」を説明できるまで読む。
5. **最小限だけ直す。** 周りの書き方・命名に合わせる。自己流リファクタや無関係な変更を混ぜない。
6. **全テスト緑を確認** 対象バグの修正とデグレが起きていないことを確かめるため。
7. **コミット・プッシュ** 修正をgitの履歴に残す。
8. **PRを作成する。** PRテンプレートに「何を変えたか / どう確認したか / 影響範囲」を書く。
9. **レビュー依頼** 解析内容、修正の妥当性を講師がチェック。

## 5. 課題セッション一覧

難易度の目安: ★☆☆ 軽い / ★★☆ 標準 / ★★★ 歯ごたえあり。課題A1 から順に進めるのがおすすめです（A＝バグ修正 / B＝仕様変更 / C＝機能追加。番号はカタログ・ブランチ名と共通のキー）。各課題の「直す場所の答え」は載せていません — **探すのが練習**です。

各バグ課題（A1〜A8）の詳細は `現象 → 再現手順 → 期待結果 → 備考・ヒント` のフォーマットで [`challenges/`](challenges/) に1ファイルずつ用意しています。**タイトルから各詳細ファイルへ飛んでください。**

| 課題 | 難易度 | タイトル | セッション |
| --- | --- | --- | --- |
| [A1](challenges/A1.md) | ★★☆ | 10%オフのクーポンがほとんど効かない | 1 金額のバグ |
| [A2](challenges/A2.md) | ★☆☆ | 在庫ちょうどの数を注文できない | 2 在庫と注文 |
| [A3](challenges/A3.md) | ★★★ | 昔の注文の金額が、商品の値上げで変わってしまう | 2 在庫と注文 |
| [A4](challenges/A4.md) | ★☆☆ | キャンセルしても在庫が戻らない | 3 キャンセルと状態 |
| [A5](challenges/A5.md) | ★★☆ | キャンセル済みをもう一度キャンセルできてしまう | 3 キャンセルと状態 |
| [A6](challenges/A6.md) | ★★☆ | 存在しない注文IDを開くと「サーバーエラー」になる | 4 エラーの扱い |
| [A7](challenges/A7.md) | ★★☆ | 合計金額が1円ずれる注文がある（テストが見逃している） | 5 テストの穴 |
| [A8](challenges/A8.md) | ★★☆ | 在庫を超えて注文できてしまう（条件あり） | 5 テストの穴 |

> セッション5（A7・A8）は、**`dotnet test` が全部緑なのにバグがある**パターン。**まず症状を再現 → それを捕まえるテストを自分で書いて赤くする → 直して緑にする** の順で取り組む（詳細は各ファイル参照）。

### 発展: 仕様変更・機能追加

ここからは「バグ直し」ではなく、**要件が変わった / 機能を足す**課題です。テストの更新・追加も自分で行います。講師から個別に出されます。

- **課題B1（★★☆・仕様変更）** クーポンに「○○円以上で使える」最低利用金額を追加する
- **課題B2（★★☆・仕様変更）** 送料を導入する（一定額未満は送料あり、以上は無料）
- **課題C1（★★☆・機能追加）** 商品名で検索できる API を足す（フロントに検索ボックスも）
- **課題C3（★☆☆・機能追加）** 在庫切れ商品を一覧で「売切」表示にする（フロント側だけで完結）

## 6. 困ったときのチェックリスト

- 症状を1文で言えるか？（「○○すると△△になる、本当は□□のはず」）
- `dotnet test` の赤いテストは何を期待している？ 実際の値は？
- それは Controller / Service / Repository / Middleware のどの層の話？
- 直す前に「なぜこの行でそうなるか」を説明できるか？
- 直した後、**全部の**テストが緑か？（直したつもりで別を壊していないか）
- 変更は最小限か？ 関係ない箇所まで書き換えていないか？

行き詰まったら、抱え込まず講師に相談してください。**「どこを調べて、何が分かって、何で詰まったか」**を言えると、それ自体が良い質問の練習になります。
