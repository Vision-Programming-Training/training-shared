# ミニEC 既存改修トレーニング — 受講者ガイド

この研修は **「ゼロからアプリを作る」のではなく、すでに動いている既存アプリに手を入れて直す」** 練習です。

配属先では、他人が書いた・もう動いているコードを読んで直す場面がほとんど。<br>
その地力を、小さなミニEC（ネットショップの注文API）で鍛えます。

特定の言語を覚えるのが目的ではありません。**どの現場でも効く次の力**を身につけます。

- 他人が書いた、もう動いているコードを読んで全体像をつかむ
- 「直したい挙動はどのファイル・どの層にある？」を探し当てる
- 既存の作法に合わせて、**最小限だけ**直す
- テストで「壊していないこと」を自分で担保する
- Issue / PR / レビューのチーム開発フローに乗る

## 2. 金額の計算ルール（重要）

コード確認に入る前に、本プロジェクトにおいての金額計算ルールをチェックしてください。<br>
覚えておくと原因特定が速くなります。

```
1. 税抜小計  = Σ(単価 × 数量)
2. クーポン  = 定額(◯◯円引き) または 定率(◯%引き)。引きすぎて0円未満なら0円
3. 税込合計  = (割引後) × 1.10 を、1円未満は四捨五入（.5は切り上げ）
```

例）
- ノート2 + マグ1（割引なし）= **1980円**
- Tシャツ1 + ペン3 − `WELCOME500`(500円引き) = **2695円**
- トートバッグ1 − `SALE10`(10%引き) = **1782円**

## 3. テストコードについて — 「直った」「壊してない」の目印

プログラミングは基本的に単体テストを前提に行うことが多いです。<br>
本プロジェクトにおいても例に漏れずバックエンド側の処理に単体テストを用意しています。

- テストが **緑（成功）= 仕様どおり動いている**目印<br>
→ **赤（失敗）= どこか壊れている**目印。


- 課題に取りかかったら、まずテストを実行<br>
→ **どのテストが赤いか**が、症状の正体を教えてくれることが多いです。

> **テストが赤いまま放置しない。**<br>
「自分の変更を入れた後に別のテストが赤くなっている = 今まで動いていた機能を壊している」<br>
という目安になるため、修正実装後は全件緑になっていることを必ずチェックしましょう。

## 4. 課題の進め方（毎回この順番で）

1. **課題ブランチをチェックアウトする** <br>
課題番号に応じた `challenge/...`をチェックアウトする。

2. **修正用ブランチを切る**<br>
チェックアウトしたブランチから自分専用のブランチを切り直す。<br>
課題用のブランチはみんなが使うものなので、そのまま利用しないこと。

3. **（バグの場合）バグを再現させる。**<br>
 原因解明のためにはまず再現させることが大事。

4. **テスト を走らせる。** <br>
赤いテストがあれば、その期待値（正しい値）と実際の値を比べる。

5. **原因を解析する。**<br>
いきなり書き換えない。まず「なぜでそうなるか」を説明できるまで読む。

6. **最小限だけ直す。**<br>
周りの書き方・命名に合わせる。無関係な変更を混ぜずに最小限で直すのがスマート。

7. **全テスト緑を確認**<br>
対象バグの修正とデグレが起きていないことを確かめるため。

8. **コミット・プッシュ**
修正をgitの履歴に残す。

9. **PRを作成する。**
PRテンプレートに「何を変えたか / どう確認したか / 影響範囲」を書く。<br>
ここを上手く説明できるようになっていれば、ちゃんと理解できている証拠となる。

10. **レビュー依頼**<br>
解析内容、修正の妥当性を講師がチェック。

## 5. 課題一覧

難易度の目安:<br>
★☆☆ 軽い / ★★☆ 標準 / ★★★ 歯ごたえあり。

課題A1 から順に進めるのがおすすめです（A＝バグ修正 / B＝仕様変更 / C＝機能追加）。<br>
各課題の「直す場所の答え」は載せていません — **探すのが練習**です。

各バグ課題（A1〜A8）の詳細は [`challenges/`](challenges/) に1ファイルずつ用意しています。**タイトルから各詳細ファイルへ飛んでください。**

### 発展: 仕様変更・機能追加

ここからは「バグ直し」ではなく、**要件が変わった / 機能を足す**課題です。テストの更新・追加も自分で行います。

- **課題B1（★★☆・仕様変更）** クーポンに「○○円以上で使える」最低利用金額を追加する
- **課題B2（★★☆・仕様変更）** 送料を導入する（一定額未満は送料あり、以上は無料）
- **課題C1（★★☆・機能追加）** 商品名で検索できる API を足す（フロントに検索ボックスも）
- **課題C3（★☆☆・機能追加）** 在庫切れ商品を一覧で「売切」表示にする（フロント側だけで完結）

## 6. 困ったときのチェックリスト

- 症状を1文で言えるか？（「○○すると△△になる、本当は□□のはず」）
- 赤いテストは何を期待している？ 実際の値は？
- 問題が起きているのはどの層？
- 直す前に「なぜこの行でそうなるか」を説明できるか？
- 直した後、**全部の**テストが緑か？（直したつもりで別を壊していないか）
- 変更は最小限か？ 関係ない箇所まで書き換えていないか？

行き詰まったら、抱え込まず講師に相談してください。<br>
「どこを調べて、何が分かって、何で詰まったか」を言えると、それ自体が良い質問の練習になります。
