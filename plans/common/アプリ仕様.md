# ミニEC研修アプリ 仕様（言語非依存 ＋ フロント）

> 研修用ミニ EC アプリの **「正しく動く完成形（`main`）」の言語非依存な仕様**。サーバーをどの言語・フレームワークで実装しても、ここで定める契約（**データモデル・API・業務ロジック**）を満たすこと。
>
> フロント（`training-frontend`）は全言語版で共有し、**無改修で接続できる**のが正しい状態。
>
> C# リファレンス実装の固有事項（バージョン・ディレクトリ構成・起動コマンド・DB 初期化の仕組みなど）は [../csharp/training-app-plan.md](../csharp/training-app-plan.md) を参照。課題版（どこを壊して研修課題にするか）は完成形から派生する別ブランチ／[課題カタログ.md](./課題カタログ.md) で管理し、本書では扱わない。

---

## 1. プロジェクトの目的

SES 配属前の新人・未経験寄りエンジニアに、**既存コードを読んで・探して・直す**実務感覚を身につけさせる研修教材。

参考イメージはゆめみの iOS 研修（天気アプリをセッション単位で課題化）。ただし本研修は **greenfield（ゼロから作る）ではなく brownfield（出来上がった保守）型**に振り切る点が最大の違い。

### 研修で身につけるスキル（言語非依存）

配属先の言語は読めないため、特定言語スキルではなく、どの現場でも通用する地力を狙う：

- 他人が書いた、もう動いているコードを読んで全体像をつかむ
- 「直したい挙動はどのファイル・どの層にある？」を探し当てる
- 既存の作法に合わせて、最小限だけ直す（俺流で書き換えない）
- テストと git で「壊していないこと」を担保する
- Issue / PR / レビューのチーム開発フローに乗る

## 2. アプリ題材：ミニEC（注文 API）

題材は「商品をカートに入れて注文する」ミニ EC。ドメイン説明が不要で、計算ロジック・状態遷移・バリデーションが揃い、修正系の課題を仕込みやすい。

### リポジトリ構成（フロント1個 ＋ サーバー言語ごとに別リポ）

サーバーサイドを別言語・別フレームワークに入れ替える可能性があるため、**フロントとサーバーをリポジトリ単位で分離**する。フロントは backend 非依存なので 1 リポジトリを全言語版で共有し、サーバーは言語ごとに独立リポジトリにする。

```
training-frontend           # 素の HTML/CSS/JS。全言語版で共有（下の「フロントの構成」を参照）
training-backend-csharp     # ASP.NET Core 版（リファレンス実装）… ../csharp/training-app-plan.md
（将来）training-backend-java # Spring Boot 版など
（将来）training-backend-node # NestJS 版など
```

サーバーリポジトリは言語が変わっても次を備える：3 層（Controller / Service / Repository）、データモデル（後述）、例外 → HTTP 変換の一元化、自動 API ドキュメント（Swagger/OpenAPI 相当）、起動時の DB 自動作成＋seed、テスト、CI、README/CONTRIBUTING/PR テンプレート。具体的なディレクトリ・ファイルは各言語版の実装仕様を参照。

### 3 層アーキテクチャ（責務分離）

「直したい挙動はどの層にあるか」を探す練習のため、層の役割をはっきり分ける。

- **Controller**：リクエストを受けるだけ。意図的に薄く保つ（ロジックを書かない。例外処理も書かない）。
- **Service**：業務ロジックの中心（合計金額・クーポン・在庫・注文/キャンセル）。**修正系の課題の主戦場。**
- **Repository**：DB アクセス。インターフェースで抽象化し、DI で注入する。

> 移植時は「その言語が標準的に提供する DI / 例外ハンドリングの仕組み」に置き換えてよい（例：Spring の `@Service`/`@RestControllerAdvice`、Nest の Provider/Exception Filter）。**3層の責務分離と「Controller を薄く保つ」方針は維持する。**

## 3. フロントエンド（全言語版で共有）

### 方針（重要）

- フロントは**作り込まない**。素の HTML/CSS/JS のみで、商品閲覧 → カート → 注文を中心に、注文履歴・管理（価格変更）まで触れる程度の最小構成。
- `fetch("http://localhost:xxxx/api/...")` で backend を叩くだけにする。**この `fetch` の行＝クラサバの境界**であり、これが見えることが狙い。
- **フレームワーク（React / Vue / Blazor など）は使わない**。とくに Blazor Server はサーバー処理がメソッド呼び出しに偽装され、境界がコードから消えるため、本研修の狙いと相反する。型共有・tRPC 的な「境界を隠す」仕組みも使わない。
- フロントは backend の言語に依存しない。接続先サーバーは `js/core/config.js` の `API_BASE_URL` で切り替える。サーバーを別言語版に入れ替えても、フロントはこの 1 行を変えるだけで再利用できる。

### 構成

```
training-frontend
├── index.html              店舗ページ
├── orders.html             注文履歴ページ
├── admin.html              管理ページ（価格変更）
├── css/                    common / shop / orders / admin
├── js/
│   ├── core/               DOM 非依存の土台
│   │   ├── config.js         接続先サーバーのベース URL（ここだけ切り替える）
│   │   ├── api.js            fetch で API を叩く（クライアントとサーバーの境界）
│   │   └── cart.js           カートの状態管理（純粋なロジック）
│   ├── ui/                 画面描画（DOM 操作）
│   │   ├── render.js         描画ヘルパ（yen / escapeHtml / 各種 render）
│   │   └── components/topbar.js  共通ヘッダーナビ
│   └── entries/            各 HTML の入口（画面と 1 対 1）
│       ├── shop.js           店舗（商品一覧・カート・注文）
│       ├── orders.js         注文履歴（一覧・キャンセル）
│       └── admin.js          管理（商品マスタ参照・価格変更）
└── README.md
```

依存の向きは `entries/* → core/api・core/cart・ui/render → core/config` の一方通行。`core/cart.js` と `ui/render.js` は互いを知らず、つなぎ役の `entries/*.js` がハンドラを渡す（関心の分離）。

## 5. API 設計

| メソッド | パス | 概要 |
|---|---|---|
| GET | `/api/products` | 商品一覧 |
| GET | `/api/products/{id}` | 商品詳細（存在しなければ 404） |
| PUT | `/api/products/{id}/price` | 商品の価格変更（存在しなければ 404。**既存注文の明細には影響しない**） |
| GET | `/api/coupons` | クーポン一覧（コード・割引種別・割引値） |
| GET | `/api/orders` | 注文一覧（明細・商品名・クーポン込み。作成日時の降順） |
| GET | `/api/orders/{id}` | 注文詳細（存在しなければ 404） |
| POST | `/api/orders` | 注文作成（商品 ID・数量・任意のクーポンコード） |
| POST | `/api/orders/{id}/cancel` | 注文キャンセル（在庫を戻し、ステータスを Cancelled に） |

### 入出力の形（DTO）

エンティティを直接返さず、DTO に詰め替えて返す。**フィールド名・大文字小文字・null 許容はフロントが依存するため厳守。**

- **ProductDto**: `{ id, name, price, stock }`
- **CouponDto**: `{ code, discountType, discountValue }`
- **OrderDto**: `{ id, status, totalAmount, couponCode, createdAt, items[] }`
- **OrderItemDto**: `{ productId, productName, quantity, unitPrice, lineTotal }`（`lineTotal = unitPrice × quantity`）
- **CreateOrderRequest**（入力）: `{ items: [{ productId, quantity }], couponCode? }`
- **UpdateProductPriceRequest**（入力）: `{ price }`
- `status` は文字列で `"Pending" | "Confirmed" | "Cancelled"`。

### 注文作成リクエストの例

```http
POST /api/orders
Content-Type: application/json

{
  "items": [
    { "productId": 1, "quantity": 2 },
    { "productId": 3, "quantity": 1 }
  ],
  "couponCode": "WELCOME500"
}
```

### エラー応答

例外は Controller では握らず、**横断的なハンドラ（ミドルウェア相当）で HTTP ステータスへ変換**する。レスポンスボディは JSON `{ "status": <code>, "error": "<message>" }`、`Content-Type: application/json`。

| ケース | ステータス |
|---|---|
| 存在しない商品 ID / 注文 ID | 404 |
| 在庫不足・無効なクーポンコード・空の注文・数量 0 以下・既にキャンセル済み | 400 |
| 入力バリデーション違反（必須・範囲）| 400 |
| 想定外の例外 | 500（メッセージは伏せる） |

## 6. ビジネスロジック仕様（移植時に最も重要）

ここが「正しい挙動の基準」。別言語でも**同じ入力に同じ出力**を返すこと。テスト（セクション 7）がこの仕様を固定している。

### 6-1. 金額計算

3 ステップで計算する。通貨は整数円。**浮動小数の丸め癖を避け、固定小数点（decimal 相当）で計算する。**

1. **税抜小計** = Σ(`unitPrice` × `quantity`)
2. **クーポン割引適用**（小計に対して）
   - `FixedAmount`：`小計 − discountValue`
   - `Percentage`：`小計 × (1 − discountValue/100)`
   - クーポンが無ければ小計のまま
   - **割引後が 0 円未満になっても 0 円で下げ止める（0 クランプ）**
3. **税込合計** = `割引後 × 1.10`（消費税 10%）を **円未満四捨五入**（`0.5` は切り上げ＝AwayFromZero。**銀行丸め＝偶数丸めは禁止**）

検証値：
- 小計 1800、クーポンなし → 1800 × 1.10 = **1980**
- 小計 2950、`WELCOME500`（定額500）→ (2950 − 500) × 1.10 = **2695**
- 小計 1800、`SALE10`（定率10%）→ 1800 × 0.9 × 1.10 = **1782**
- 小計 105、クーポンなし → 115.5 → 四捨五入で **116**

### 6-2. 注文作成

- 明細が 0 件なら 400（「商品を 1 つ以上」）。
- **同じ商品が複数行で来たら数量を合算してから在庫チェック**する（合算後の数量が在庫を超えれば 400）。
- 数量が 1 以上であること（0 以下は 400）。入力バリデーションでも弾くが、Service でも担保する。
- 商品 ID が存在しなければ 404。
- 各商品について `在庫 < 要求数量` なら 400（在庫不足。メッセージに商品名・在庫・要求数を含める）。
- 在庫を要求数量だけ減らす。
- 明細の `unitPrice` には**その時点の商品価格をスナップショット**として保存する。
- クーポンコードが指定されていれば該当クーポンを引く。**該当しなければ 400**（無効なクーポン）。空文字・未指定はクーポンなし扱い。
- `totalAmount` を 6-1 で確定し、ステータス `Confirmed`・`createdAt`（UTC）で保存する。
- 保存後、商品名・クーポン込みの完全なグラフを読み直して `OrderDto` を返す。

### 6-3. 注文キャンセル

- 対象注文が無ければ 404。
- **既に `Cancelled` なら 400**（二重キャンセル禁止）。
- 明細の数量だけ各商品の在庫を戻す。
- ステータスを `Cancelled` にして保存する。

### 6-4. 価格変更

- 対象商品が無ければ 404。
- 商品マスタの `price` を更新する。
- **過去の確定済み注文の明細（`unitPrice`・小計・合計）は変わらない**（6-2 のスナップショットにより担保される）。これは「履歴は注文時点の値で残る」を見せる教材ポイント。

### 6-5. 一覧取得

- 注文一覧は **作成日時の降順**（新しい順）。
- 一覧・詳細とも、明細・商品名・クーポンを **まとめて読み込んで N+1 を避ける**（eager loading 相当）。

## 7. テスト観点

主要なロジックを**ユニットテストで固定**し、全てグリーンにする。これが「正しい挙動の保証」であり、課題版を切り出すときの解答にもなる。移植先でも同等のケースを用意する（テストフレームワークは各言語の定番でよい）。

**金額計算**
- 小計 = Σ(単価 × 数量)
- クーポン無しは小計のまま / 定額を引く / 定率を掛ける
- 税込（×1.10）/ クーポンは税の前に適用 / 円未満四捨五入

**注文・在庫・キャンセル**
- 注文作成で在庫が減り、合計が正しく、明細（商品名・単価スナップショット・小計）が入る
- 在庫ぴったりの数量で成功する（境界）
- クーポン適用で合計が変わる
- 在庫不足で 400（かつ在庫が変わらない）
- 存在しない商品で 404
- 無効なクーポンで 400
- キャンセルで在庫が戻り `Cancelled` になる / 二重キャンセルで 400 / 不明な注文のキャンセルで 404
- 不明な注文の取得で 404
- 一覧は新しい順
- **商品価格を後から変えても、確定済み注文の単価・小計・合計は不変**（スナップショット）

## 8. 環境構築・CORS（考え方）

新人が初日につまずかないことを最優先する。具体的なコマンドは各言語版の実装仕様を参照。

### セットアップ（2 リポジトリを clone）

受講者は「サーバー」と「フロント」の 2 リポジトリを clone する。

- **サーバー**：1 コマンドの起動で「依存解決 → ビルド → 起動 → DB 作成 → seed」まで自動で完了すること。DB はインストール不要のファイル DB（SQLite 相当）。DB ファイルを消して再起動すれば初期データに戻る。
- 起動後、自動生成された API ドキュメント（Swagger/OpenAPI 相当）でフロント無しでも API を叩いて確認できる。
- **フロント**：`index.html` を簡易 HTTP サーバー（VS Code Live Server / `python -m http.server` など）で開く。`js/core/config.js` の `API_BASE_URL` を起動中サーバーのベース URL に合わせる。

### CORS について（リポジトリ分離ゆえに必須）

フロント（例: `http://localhost:5500`）とサーバー（別ポート）はオリジンが異なるため、何もしないと `fetch` がブラウザにブロックされる。**サーバー側にフロントのオリジンを許可する CORS ポリシーを設定済みの状態で渡す**こと（ヘッダ・メソッドは全許可相当）。許可オリジンは設定に切り出し、追記しやすくする。

README に「なぜこの設定が要るのか（フロントとサーバーが別オリジンだから）」を一言だけ添える。新人が概念として理解できればよく、設定作業自体は課題にしない。

### リポジトリ整備

PR・CI はサーバーリポジトリ側に置く。フロントリポジトリは最小限の README のみ。

サーバーリポジトリ：
- **README.md**：このアプリの概要（ミニ EC の注文 API）、技術スタック（バージョン固定）、2 リポジトリのセットアップ手順、コマンド一覧、CORS の一言説明、3 層構成の説明、API 一覧、seed の内容
- **CONTRIBUTING.md**：ブランチ命名規則、コミット・PR の作法
- **PR テンプレート**：何を変えたか／どう確認したか／影響範囲
- **CI**：push / PR でビルド＋テストを自動実行。グリーンを目印にする

フロントリポジトリ：
- **README.md**：何のためのフロントか、`js/core/config.js` で接続先サーバーを切り替える方法、簡易 HTTP サーバーでの起動手順、ファイル構成と依存の向き

## 9. 実装方針

- `main` は**正しく動く完成形**。バグを仕込んだり、わざと未実装にしたりしない。全テストがグリーンの状態で完成とする。
- 3 層（Controller / Service / Repository）の責務分離を守る。Controller は薄く、ロジックは Service に置く。
- コードは普通に読みやすく書く（新人が読んで追える構成・命名）。将来この完成形が課題版の基準になるため、まずは綺麗な完成形を優先する。

## 10. 横展開（別言語への移植）

SES で配属先が読めないため、本教材は 1 言語で完成させた後、他言語へ横展開できる構造にしてある。リポジトリを分けたことで、横展開がそのまま実現できる。

- **サーバーは言語ごとに別リポジトリ**（`training-backend-csharp` / `-java` / `-node` …）。入れ替え＝別リポジトリを使うだけで、履歴も依存も完全に分離される。
- **フロントは `training-frontend` の 1 個を全言語版で共有**。`js/core/config.js` の `API_BASE_URL` を切り替えるだけで、どのサーバーにも接続できる。新しいサーバー版を作ってもフロントは作り直さない。
- 移植で守るべき不変点：**データモデル（セクション 4）/ API 仕様（セクション 5）/ ビジネスロジック（セクション 6）/ テスト観点（セクション 7）**。言語固有の置き換えが許されるのは DI・例外ハンドリング・ORM・丸め API など実装手段の部分のみ。
- 各サーバー版は、同じ入力に同じレスポンス・同じステータスコードを返すこと。フロントはサーバー言語を問わず無改修で動くのが正しい状態。
