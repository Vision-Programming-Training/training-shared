# ミニEC 研修課題 カタログ（言語非依存 / 講師・教材管理者用）

> このカタログは**言語に依存しない**。狙い・受講者に見える症状・壊している不変条件・難度・セッション構成だけを定義する。
> 各言語版の「実際の仕込み diff・該当ファイル・テスト名」は、**`training-shared/plans/<言語>/` のドキュメント**に置く（C# 版は `training-shared/plans/csharp/課題-仕込みdiff-csharp.md`）。
> **取り扱い注意**: 各課題の「壊している不変条件」は解答そのもの。受講者には渡さない。受講者には `受講者ガイド.md` と課題ブランチだけを渡す。

---

## 0. このドキュメントの位置づけ

`training-app-plan.md` が「完成形ができてから別工程で検討する」としていた**課題化フェーズ**のカタログ。各言語版バックエンド（`training-backend-csharp` / 将来の `-java` / `-node` / `-go` …）の `main`（正しく動く完成形＝解答の基準）を出発点に、**どこを壊す / 変える / 足す**かを定義する。

このカタログが言語非依存でいられるのは、`backend-port-design` スキルが抽出する「**言語非依存の契約**（API・JSON 形・金額計算・業務ルール・エラー応答）」に課題を紐づけているから。契約が同じである限り、**狙い・症状・不変条件は全言語版で成り立つ**。一方で「それをどのコードで壊すか」は実装の都合なので各言語側に分離する。

```
training-shared/plans/common/課題カタログ.md          ← 本書。言語非依存（狙い・症状・不変条件・難度）
training-shared/guides/受講者ガイド.md                ← 言語非依存（起動/テストコマンドだけ各言語追記）
training-shared/guides/challenges/A1.md … A8.md       ← 受講者向け各課題ガイド（言語非依存）
training-shared/plans/csharp/課題-仕込みdiff-csharp.md ← C# 固有：実 diff・テスト名・file path
（将来）training-shared/plans/java/課題-仕込みdiff-java.md ← Java 版の diff を別途起こす
```

### 課題の3タイプ

| タイプ | やること | 担保のしかた |
|---|---|---|
| **A バグ修正型** | `main` のコードを一箇所壊したブランチを渡し、症状から原因を特定して直させる | 既存テストが**赤**になる／直すと**緑**に戻る |
| **B 仕様変更型** | 「要件が変わった」体で既存挙動の変更を依頼。テストも更新させる | 新しい期待値のテストを受講者が書き換え or 追加して緑 |
| **C 機能追加型** | 新エンドポイント／フロント連動を、既存の3層の作法に合わせて足させる | 追加テスト緑＋フロントで動作確認 |
| **D 横断型** | N+1・CORS・PRレビューなどチーム開発スキル | 体感・レビュー観点で評価 |

### ブランチ運用（推奨・言語共通）

```
main                         # 正しい完成形（解答）。触らない
challenge/a1-coupon          # A1（受講者ガイドの課題1）を仕込んだ配布用ブランチ
challenge/a2-offbyone        # A2（課題2）…
solution/a1-coupon           # （任意）解説用に main との diff を残す
```

受講者は `challenge/*` を checkout して着手 →（CONTRIBUTING の作法で）ブランチを切って修正 → PR。CI（各言語のテスト実行）の緑が目印。各課題は `main` から1つだけ仕込む（複数同時に仕込まない）。

## A. バグ修正型（研修の主戦場）

各課題は **狙い / 受講者に見える症状 / 壊している不変条件（＝解答の核）/ 検証値 / 難度** で定義する。実コードの diff は各言語の仕込み diff 集を参照。

> **番号のキー**: A1〜A8 の番号は**受講者ガイドの課題1〜8（進める順）**と一致させてある。受講者ガイド／各言語の仕込み diff 集／ブランチ名すべてで、同じ番号が同じ課題を指す。

### A1. 定率クーポンの計算ミス ★★☆
- **狙い**: 割引種別の分岐（定額 / 定率）の取り違えと、割引式の追跡。
- **症状**: `SALE10`（10%オフ）を使うとほとんど割り引かれない。`WELCOME500`（定額）は正常。
- **壊している不変条件**: 定率は `小計 ×(1 − 値/100)`。定額の式（`小計 − 値`）と取り違えると破れる。
- **検証値**: トートバッグ1点(1800) − `SALE10` → 1782 円。

### A2. 在庫チェックの境界（off-by-one）★☆☆
- **狙い**: 境界値・比較演算子1文字。
- **症状**: 在庫ぴったりの数量を注文すると「在庫不足」で弾かれる。
- **壊している不変条件**: `在庫 < 要求数量` のときだけ不足。`<=` にすると在庫ちょうどが買えない。
- **検証値**: 在庫50のノートを50個 → 注文成功・在庫0（完成形の境界テストもこのケースを検証）。

### A3. 単価スナップショットの崩れ ★★★
- **狙い**: 「金額はいつの時点の値か」。注文時点の単価が保存される設計意図を読み取る。
- **症状**: 商品価格を変更すると、**過去の確定注文の明細単価・小計まで変わって見える**。注文の保存済み合計と食い違う。
- **壊している不変条件**: 明細単価は注文時点のスナップショット。表示・再計算で現在価格を参照すると破れる。
- **検証値**: 注文後にノート価格を 300→9999 に変更しても、その注文明細は 300 のまま。

### A4. キャンセルで在庫が戻らない ★☆☆
- **狙い**: 状態遷移に伴う副作用（在庫の戻し）。
- **症状**: キャンセルでステータスは Cancelled になるが在庫が増えない。
- **壊している不変条件**: キャンセル時に明細数量ぶん在庫を戻す。
- **検証値**: 在庫50→5個注文で45→キャンセルで50に戻る。

### A5. 無効なクーポンコードが黙って無視される（ガード節の欠落）★★☆
- **狙い**: ガード節の意味と「ルックアップ結果の `null` を握りつぶさない」防御的プログラミング。クーポンコードは利用者の自由入力なので、検証はサーバーの責務。
- **症状**: 存在しないクーポンコード（例 `SALE99`）を入れて注文すると、エラーにならず**割引なしの満額で注文が通ってしまう**。利用者は割引したつもりで満額を請求される。有効なコードはこれまで通り効く。
- **壊している不変条件**: コードで検索して見つからなければ 400 で弾く。`null`（＝見つからない）をそのまま「クーポン未使用」として処理に流すと破れる。
- **検証値**: トートバッグ1点(1800) + `SALE99`（無効）→ 400 エラー。`SALE10`（有効）→ 1782 円で成功。

### A6. NotFound が 500 になる ★★☆
- **狙い**: 「例外/エラー → HTTP ステータス」変換の集約点を探す力。
- **症状**: 存在しない注文IDを開くと 404 ではなく 500（メッセージも伏せられる）。
- **壊している不変条件**: 「見つからない」は 404。NotFound と 500 の対応付けを壊すと破れる。
- ⚠️ **集約点が FW で大きく変わる課題**（後述）。

### A7. 税込の丸めが狂う ★★☆
- **狙い**: 丸め仕様（四捨五入 / 偶数丸め / 切り捨て）の違いを意識させる。**既存テストが見逃しているケースを自分で再現テスト化する**力も問う。
- **症状**: 多くの注文は合計が合うのに、税込が「.5 ちょうど・かつ端数処理で偶数側に倒れる」注文だけ 1 円ずれる。`dotnet test` などは全部緑のまま（既存テストが当該ケースを試していない）。
- **壊している不変条件**: 税込合計は「1円未満を四捨五入（.5 は切り上げ＝AwayFromZero）」。偶数丸め/切り捨てにすると破れる。
- **検証値**: ボールペン(150) + `SALE10`（→135→税込 148.5）→ 正解 149 円（偶数丸めだと 148）。※商品価格が全て 50 の倍数のため、端数は `SALE10` 併用時のみ発生（クーポン無し/定額では再現しない）。
- **テスト**: 完成形テストはこのケースを試しておらず**素通り（赤にならない）**。**まず再現テストを書く**工程を含める。
- ⚠️ **言語差が大きい課題**（後述「移植時の注意」参照）。

### A8. 割引で請求がマイナスになる（0円下限の欠落）★★☆
- **狙い**: 「再現条件を自分で作る」力と、下限ガード（クランプ）。一見動くが、特定の入力でだけ破綻するバグ。
- **症状**: 普通の注文は問題なし。だが**小計より大きい定額クーポン**を使うと、合計が0円を下回り**マイナスの金額で注文が通る**。
- **壊している不変条件**: 割引後が0円未満になっても請求は**0円までにクランプ**する（受講者ガイド §2 の計算ルールに明記）。
- **検証値**: ノート1点(300) + `WELCOME500`（定額500引き）→ 割引後 −200 を 0 にクランプ → 税込 0 円。バグだと (300−500)×1.10 = −220 円。
- **テスト**: 完成形（`main`）のコードはクランプを実装しているが、それを検証するテストは「初期テストにこのケースが漏れていた」想定で**完成形からあえて外してある**（A7 と同じ進め方）。配布ブランチでは加えてクランプ処理そのものを外す（テストは元から無いので緑のまま）。**まず再現テストを書く**工程を含める。

## B. 仕様変更型（要件変更を既存の作法で反映）

> 「正解」が一通りではない。要件と不変条件を示し、テスト更新も含めて受講者に依頼する。

### B1. クーポンに最低利用金額 ★★☆
- **要件**: 「`SALE10` は税抜小計 3000 円以上でのみ使用可。未満なら 400」。
- **不変条件の追加**: クーポンに最低利用金額属性。未満なら適用不可エラー。
- **学び**: モデル変更が seed・検証・テストへ波及することの体感。

### B2. 送料の導入 ★★☆
- **要件**: 「税抜小計 5000 円未満は送料 500 円。5000 円以上は無料。送料は税込合計に加算（非課税）」。
- **不変条件の追加**: 計算順序を明文化（税計算 → 丸め → 送料加算が無難）。既存の検証値が変わる。
- **学び**: 計算順序と検証値の同時更新。回帰テストの読み替え。

### B3. 1商品あたりの注文上限 ★☆☆
- **要件**: 「同一商品は合算 99 個まで。超過は 400」。
- **不変条件の追加**: 合算後数量の上限チェック。既存の空・0以下チェックと同じ場所・同じ作法で。
- **学び**: 既存バリデーションに揃えて足す。

### B4. 商品ごとの軽減税率 ★★★
- **要件**: 「食品カテゴリは 8%、それ以外 10%」。
- **不変条件の変更**: 「全体一律10%」前提が崩れ、明細ごとに税率適用 → 合算へ構造が変わる。
- **学び**: 前提が崩れると計算ロジックの構造ごと変わる波及の大きさ。上級者向け。

## C. 機能追加型（新エンドポイント / フロント連動）

### C1. 商品検索 ★★☆
- **要件**: `GET /api/products?keyword=...` で名前部分一致。空なら全件（既存挙動維持）。
- **触る範囲**: Controller（薄く）/ Repository（部分一致）/ Service。フロントに検索ボックス。
- **テスト観点**: keyword 一致 / 不一致 / 空。

### C2. 注文ステータス更新 API ★★☆
- **要件**: `POST /api/orders/{id}/confirm` で `Pending → Confirmed`。`Cancelled` からは不可（400）。
- **注意**: 完成形は作成時 `Confirmed` 固定で `Pending` を作る経路が無い。「作成時を `Pending` に」とセットにするか割り切るかは設計判断として受講者に委ねる。
- **学び**: 状態遷移の許可/不許可を表で書いてから実装する。

### C3. 在庫切れの表示（フロント主体）★☆☆
- **要件**: 在庫 0 の商品（seed のステッカー）を一覧で「売切」表示にし、カート追加を不可に。
- **触る範囲**: **フロント（`training-frontend`）のみ**。API は既に `stock` を返すのでバックエンド改修不要。
- ✅ **全言語版で完全に同一**（フロント共有のため、どのバックエンドに繋いでも同じ課題が成立）。
- **学び**: 「サーバーは既にデータを返している。表示はフロントの責務」という境界の理解。

## D. 横断・チーム開発型

### D1. N+1 を体感する ★★☆
- **仕込み**: 注文一覧の取得で関連（明細・商品・クーポン）の一括読み込みをやめ、件数ぶん追加クエリを発生させる。
- **症状**: 注文件数ぶん `SELECT` が増える。SQL ログで本数を観察させる。
- **学び**: 一覧取得の N+1 という普遍的な落とし穴。
- ⚠️ **ORM 前提の課題**（後述。raw SQL 版では自然には存在しない）。

### D2. CORS 事故の切り分け ★☆☆
- **仕込み**: CORS 許可オリジンを実フロントと別ポートにする／許可を外す。
- **症状**: フロントから fetch がブラウザにブロックされる。サーバー単体（Swagger 等）では動く。
- **学び**: 「サーバーは正常、繋ぎ目が原因」の切り分け。`training-app-plan.md` の CORS の意義を実地で理解。
- ✅ **ほぼ完全に移植可**（CORS は HTTP レベルでどの FW でも設定する）。

### D3. PR レビュー演習（俺流リファクタを直す）★★☆
- **題材**: 「動くが作法に反する」PR を用意（Controller に業務ロジック / 例外を握りつぶし / 命名規約違反）。
- **やること**: 受講者にレビュアーをやらせ、「最小修正・既存の作法に合わせる」観点で指摘 → 直す。
- **学び**: `CONTRIBUTING.md` / PR テンプレートの観点と3層責務の遵守。
- ✅ **完全に移植可**（規約・層の話で言語非依存）。

## 移植時の注意（言語版を起こすとき必読）

狙い・症状・不変条件は全言語で成り立つが、**「何が自然な正解で、何が罠か」が言語で入れ替わる課題**がある。新言語版の仕込み diff を書くときは、その言語の標準挙動を確認してから仕込むこと。

### A7 丸め — 言語ごとに既定の罠が違う（最重要）
不変条件は同じ（`115.5 → 116`、AwayFromZero／半数切り上げ）。だが「素直に書いたコード」が正解か罠かは言語で逆転する。

| 言語 | 標準の丸め | 仕込みの考え方 |
|---|---|---|
| C# | `Math.Round` 既定は**偶数丸め** | 完成形は `AwayFromZero` を明示。それを既定（偶数丸め）に戻すのが仕込み |
| Java | `Math.round` は**半数切り上げ**（正の値で 116） | 素直だと正しい。`(int)` キャスト＝切り捨てや `BigDecimal` の `HALF_EVEN` で仕込む |
| JavaScript/TS | `Math.round` は**半数切り上げ**。ただし浮動小数誤差 | 切り捨て(`Math.floor`)や `toFixed` の罠で仕込む。金額は整数/固定小数で扱う前提を崩さない |
| Python | `round()` は**偶数丸め** | 素直だと罠。完成形は `Decimal` + `ROUND_HALF_UP`。それを素の `round()` に戻すのが仕込み |

> ポイント: 各言語の完成形（`main`）が**まず正しい丸め**になっていること（契約遵守）を担保し、その上で「既定の罠に戻す」のを仕込みにすると、言語横断で同じ学びになる。

### A6 例外→HTTP — 集約点を探す場所が変わる
不変条件（NotFound は 404）は同じだが、「どこで変換しているか」が FW で違う。ヒントの文言を各言語に合わせる。

| FW | 集約点 |
|---|---|
| ASP.NET Core | 例外ハンドリング Middleware |
| Spring Boot | `@ControllerAdvice` / `@ExceptionHandler` |
| NestJS | Exception Filter |
| FastAPI | `exception_handler` |
| Go (net/http) | 明示的な error 判定 → ステータス書き分け（集約ヘルパ） |

### D1 N+1 — ORM 前提
ORM（EF / JPA / Prisma / TypeORM）版では「一括読み込みを外す」で自然に再現できる。**raw SQL 版（Go + sqlc 等）では N+1 は自然には起きない**ので、意図的にループ内クエリを書く形にするか、この課題は出題しない判断もあり。

### A8 0円クランプ — 移植容易
仕込みは「割引後を 0 で下限クランプする処理を外す」で全言語共通（`max(0, discounted)` 相当を消すだけ）。`min`/`max` やゼロ比較のイディオム差はあれど考え方は同じ。クーポンコードはフロントの自由入力なので、「安い商品＋小計超えの定額クーポン」で UI から再現できる。

### その他（移植容易）
A1 / A2 / A3 / A4 / A5 は四則演算・比較・代入・ガード節の話で、イディオムの違いはあれど**仕込みの考え方はそのまま移植できる**。B 系・C 系も要件は言語非依存（触る層が各言語の構成に対応するだけ）。

## 付録: 課題と既存テストの対応（言語非依存の観点）

| 課題 | 完成形テストが赤になるか | 追加テスト要否 |
|---|---|---|
| A1 | 定率クーポンのテストで赤 | 不要 |
| A2 | 在庫ちょうどの境界テストが完成形にあり、`<=` にすると赤 | 不要 |
| A3 | 注文後の価格変更テストが完成形にあり赤 | 不要 |
| A4 | キャンセルで在庫が戻るテストで赤 | 不要 |
| A5 | 無効クーポンのテストで赤 | 不要 |
| A6 | NotFound のテストで赤（仕込み方による） | 仕込み方による |
| A7 | 既存テストは端数の出ない金額で赤にならない | **要**（再現テスト） |
| A8 | 完成形にクランプ検証テストが無いため赤にならない | **要**（再現テスト） |
| B1–B4 | 期待値が変わる | **要更新** |
| C1–C3 | — | **要追加** |
| D1 | — | （観察課題） |

> 完成形テストが赤にならない課題（C# 版では A7/A8）は、「**まず再現テストを書く**」工程を課題に含めると brownfield の実務感に近い。これは全言語共通の方針（どの課題が該当するかは各言語の完成形テストの網羅状況による。C# 版では A2 の境界テストは完成形に追加済みなので該当しない）。
